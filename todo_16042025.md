# TODO for Codex: Organizational Multi-Agent Dataset Analysis System

## Goal
Build an organizational-style multi-agent system for dataset understanding, tool creation, testing, and domain-guided ECG data preparation. The system should be modular, auditable, and easy to extend.

## High-Level Architecture

### 1. Organization Head
- Implement a top-level coordinator agent.
- Responsibilities:
  - receive user goals
  - break work into department-level tasks
  - route requests between departments
  - approve major outputs before they are considered complete
  - ensure all communication between departments goes through the Organization Head

### 2. Department Structure
Each department must have:
- one **Department Head agent**
- one or more **Worker agents**
- a clear contract for inputs, outputs, and reporting

Department heads communicate upward to the Organization Head and downward to worker agents.
Worker agents should not communicate directly with other departments.

## Department 1: Knowledge Department
Purpose: collect reusable knowledge about datasets from provided links and documents.

### Inputs
- dataset landing page links
- data paper links
- documentation links
- any other relevant dataset-related URLs

### Responsibilities
- scrape and parse dataset-related web resources
- extract structured knowledge such as:
  - dataset name
  - modality
  - source
  - tasks/labels
  - metadata format
  - file organization
  - patient/study counts when available
  - splits if officially provided
  - license/usage restrictions
  - citation information
  - known limitations
- generate reusable knowledge files

### Required implementation tasks
- Use **Scrapling** as a primary web scraping utility for collecting structured information from dataset pages and documentation. The library provides adaptive parsing, CLI support, and optional shell/AI features.   - fetch HTML pages
  - extract title, main content blocks, tables, links, and metadata
  - normalize extracted content into JSON/Markdown knowledge files
  - save raw source URLs alongside processed summaries
- Prefer reusable knowledge artifacts such as:
  - `knowledge/datasets/<dataset_name>/overview.json`
  - `knowledge/datasets/<dataset_name>/references.json`

## Department 2: Coding Department
Purpose: write Python tools that analyze datasets by taking a dataset directory as input.

### Responsibilities
- create Python tools that can be executed from the command line
- tools must accept a dataset directory as input
- tools must produce outputs in structured formats such as JSON, CSV, Markdown, or plots when needed
- request dataset knowledge from the Knowledge Department when needed

### Output expectations
The main output of this department is a toolset that can:
- inspect folder structure
- summarize files and metadata
- generate statistics
- identify label distributions
- validate expected formats
- prepare split suggestions or data summaries for downstream ML work

### Constraints
- tools must not hardcode a specific dataset path
- tools must be reusable across datasets where possible
- every tool must have:
  - a clear CLI interface
  - help text
  - documented inputs/outputs
  - predictable file naming

### Suggested technical tasks
- [ ] create `coding_department/` package
- [ ] define tool contracts for all analysis scripts
- [ ] implement CLI entry points
- [ ] output machine-readable JSON summaries
- [ ] output CSV statistics tables where relevant
- [ ] generate Markdown reports for humans
- [ ] add configuration support for dataset-specific logic
- [ ] document how knowledge files are consumed by coding tools

## Department 3: Testing Department
Purpose: validate quality, correctness, reproducibility, and usability of the tools produced by the Coding Department.

### Responsibilities
- run scripts step by step
- validate outputs against expectations
- report bugs and improvement suggestions
- re-run after fixes
- produce a final test report when tools pass

### Required testing behavior
- test each script independently
- test small subsets before full runs
- verify:
  - CLI usability
  - output existence
  - output schema correctness
  - reproducibility on repeated runs
  - error handling for missing/invalid paths
- produce a structured report for each tested script

- [ ] implement smoke tests for all tools
- [ ] implement regression tests for output formats
- [ ] create sample miniature dataset fixtures
- [ ] create automated test reports in Markdown/JSON
- [ ] add a pass/fail checklist per tool

## Department 4: Cardiology Department
Purpose: provide domain expertise for ECG dataset filtering, quality control, and ML-relevant splitting strategies.

### Responsibilities

- suggest filtering rules for low-quality or problematic ECGs
- suggest clinically meaningful dataset splitting strategies

### Expected outputs

- recommendations for dataset partitioning

## Communication Rules
- All department-to-department communication must go through the Organization Head.
- Department Heads communicate with the Organization Head.
- Worker agents communicate only with their own Department Head.
- Cross-department requests must be logged.

### Required implementation tasks
- [ ] define message schema for agent communication
- [ ] log task delegation and responses
- [ ] keep approval status for each major artifact

---

## Execution Workflow
1. User gives goal to Organization Head.
2. Organization Head delegates dataset understanding to Knowledge Department.
3. Knowledge Department scrapes sources and prepares reusable knowledge files.
4. Coding Department uses the knowledge files to build dataset-analysis tools.
5. Cardiology Department reviews ECG-specific assumptions and rules.
6. Testing Department runs and validates tools step by step.
7. Organization Head reviews outputs and decides whether the system is ready.

## Engineering Requirements
- Use Python for core tools.
- Keep modules small and reusable.
- Favor typed functions and dataclasses/pydantic models where useful.
- Store intermediate and final outputs in well-defined locations.
- Add logging to all major scripts.
- Provide step-by-step execution instructions.
- Test on a small subset before any full dataset run.
- Require explicit user confirmation before running expensive or full-scale processing.
